C++ finally gets a mirror

July 2026

For as long as C++ has existed, a program couldn't look at its own types. Given a plain struct, there was no way to ask what fields it has, and something as simple as printing an enum's name meant keeping a table by hand or generating code with macros nobody enjoys reading. Most other languages grew some way for code to see itself a long time ago, and C++ kept not having one. C++26 changes that with static reflection. Since compiler support is still settling, I've been using a pinned build of clang to find out what it feels like in practice, and I've collected what I built into metawrap, a small library on top of the new API. This post is a tour of the parts I actually use.

the new syntax

Reflection adds two operators and a loop. The first operator is ^^, which lifts a type (or a member, or an enumerator) into an ordinary value of type std::meta::info that you can pass around and compute with at compile time. Here is how metawrap gets the field list of a struct:

template <typename T> consteval auto fields_of() {
  return std::define_static_array(std::meta::nonstatic_data_members_of(
      ^^std::remove_cvref_t<T>, std::meta::access_context::current()));
}

nonstatic_data_members_of does what it says, and it takes an access context because what you're allowed to see depends on where you're asking from. define_static_array gives the resulting list static storage so it can be walked later, still at compile time.

The second operator goes the other way. [:member:] is called a splice, and it turns a reflection value back into real code. Combined with template for, which expands its body once per element at compile time, you can visit every field of any struct:

template <typename T, typename F>
constexpr void for_each_field(T &&obj, F &&fn) {
  template for (constexpr auto member : fields_of<T>()) {
    fn(std::meta::identifier_of(member), obj.[:member:]);
  }
}

Each expansion of the loop body is its own piece of code, so obj.[:member:] has the actual type of that field, and the lambda you pass in gets the field's name and a reference to its value. That one function is most of the library. Everything else is applications of it.

enums learn their own names

The enum version of the same idea is the feature I'd wanted longest. This is the whole implementation of enum to string:

template <typename E>
  requires std::is_enum_v<E>
constexpr std::string_view enum_to_string(E value) {
  template for (constexpr auto e : enumerators_of<E>()) {
    if (value == [:e:])
      return std::meta::identifier_of(e);
  }
  return {};
}

No macro, no generator, no table that drifts out of sync when someone adds an enumerator. mw::enum_to_string(LogLevel::warning) returns "warning", and the reverse direction is the same loop comparing names instead of values.

a struct that prints itself

Put field iteration and enum names together and you get the thing C++ programmers have been writing by hand forever. metawrap has a struct_formatter that plugs any struct into std::format with one line:

struct ServerConfig {
  std::string host;
  int port;
  LogLevel log_level;
  bool tls;
};

template <>
struct std::formatter<ServerConfig> : mw::struct_formatter<ServerConfig> {};

ServerConfig config{"localhost", 8080, LogLevel::info, true};
std::println("{}", config);
std::println("{:k}", config);

which prints:

{ "localhost", 8080, info, true }
{ host: "localhost", port: 8080, log_level: info, tls: true }

The formatter walks the fields, quotes the strings, and looks up enum names with the function above. The format spec letters are mine (k for keys, t for the type name, p for pretty printing), and adding one is a case in a switch.

running it today

The catch is the toolchain. Reflection made it into C++26 in 2025 but no released compiler ships it yet, gcc trunk covers most of it, upstream clang covers less, and the fullest implementation is still clang-p2996, the experimental fork that tracks the proposal. That fork is what metawrap pins in its Dockerfile, and a build looks like this:

clang++ -std=c++26 -freflection-latest -stdlib=libc++ -Iinclude examples/fields.cpp

The flag says latest because the implementation is still moving, so whatever compiles this month might need touching up next month. I wouldn't put it near production, but for playing with what's coming it works today, and the examples directory has a small runnable file per feature.

C++ took its time with this one, but now that it's here I don't plan on writing another enum table by hand. Structs in the mirror are closer than they appear.
